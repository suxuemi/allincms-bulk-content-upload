> зЁіе®ҡеҸҜеӨҚз”Ёз»“и®әе·ІеңЁ `references/source-files-to-site-package.md` + operational-findings йЎ¶йғЁ Invariants + ж ЎйӘҢеҷЁ/жөӢиҜ•гҖӮ
> дҝқз•ҷжӯӨеҺҹе§ӢжҳҺз»Ҷд»…дҫӣ**еҝ«йҖҹжҒўеӨҚ/иҖғеҸӨ**,дёҚжҳҜжҙ»и·ғжҢҮеј•гҖӮ

- Context: skill-maintenance round for the workflow where a user provides several source files and expects Codex to create a publishable AllinCMS site with pages, products, posts, and site information.
- Symptom: the existing skill had source-input requirements and posts/products manifest gates, but no explicit local artifact between raw source/wiki extraction and remote site creation/upload. Without that artifact, an operator could jump from PDF/catalog extraction straight to `/sites` creation or schema-gated upload planning without a user-confirmed content contract.
- Evidence: the new `source-files-to-site-package.md` reference and `build_source_site_package.py` / `validate_source_site_package.py` helpers define an `allincms_source_site_package` with source refs, wiki refs, site proposal, pages, products, posts, draft manifests, operation gaps, confirmation gate, blocked remote actions, and adversarial checks.
- Risk: source-derived draft copy can be mistaken for remote mutation authorization, business copy can leak into the skill package, or generated products/posts can bypass current-site schema capture because the content looks ready.
- Rule: user files must flow through source intake, raw extraction, wiki distillation, local source-site package, validation, and user confirmation before any create-site or upload stage. The package is a content contract only: manifests stay `schemaVerified: false`, remote actions stay blocked, and current-site save-request capture plus sample verification remain mandatory before upload.
- Follow-up: `SKILL.md` now routes source-file runs through the new reference and validators before site creation or batch upload.

- Context: follow-up maintenance on the same source-file workflow.
- Symptom: after adding the source-site package builder, the first two gates were still underspecified: "з”ЁжҲ·з»ҷеҮ дёӘж–Үд»¶" had no deterministic source inventory, and "жўізҗҶжҸҗзӮјжҲҗ wiki" had no sourceRef validator before package generation.
- Evidence: `build_source_inventory.py` now records source refs, types, paths, sizes, and hashes outside the skill package; `build_source_wiki.py` builds an `allincms_source_wiki` from inventory plus extraction summary; `validate_source_wiki.py` checks required site/pages/products/posts fields, source refs against inventory, raw Markdown, and sensitive strings.
- Risk: an operator could lose track of which files were in scope, cite sourceRefs that do not exist, or produce a package from unverified extraction notes. That weakens later user confirmation because the user reviews polished content without a reliable source chain.
- Rule: source-file runs must start with `allincms_source_inventory`, then `allincms_source_wiki`, then `allincms_source_site_package`. Every page/product/post should cite inventory-backed sourceRefs before the user is asked to confirm the package.
- Follow-up: `source-files-to-site-package.md` and `SKILL.md` now document the inventory/wiki/package sequence and helper commands.

- Context: skill-maintenance round for the gate after a user confirms the generated source-site package.
- Symptom: the source package had a confirmation gate, but no deterministic confirmation record or execution plan that separated "user accepts this content package" from "user authorizes this exact AllinCMS remote mutation".
- Evidence: `make_source_package_confirmation.py` and `validate_source_package_confirmation.py` now build and validate a local-only confirmation artifact with `isRemoteMutationAuthorization: false`, accepted fields, confirmed counts, and all package blocked remote actions preserved as still needing action-specific authorization. `build_confirmed_site_execution_plan.py` then creates a prepared-only stage plan with command templates that retain authorization-source placeholders.
- Risk: an operator could treat content/package confirmation as blanket permission to create a site, save pages, bind routes, upload media, publish posts/products, add domains, or replay JSON requests. The opposite risk is losing the user's content approval and asking them to re-review the same package before every later schema gate.
- Rule: package confirmation is content-intent proof only. It may unlock local execution planning, but every remote AllinCMS mutation still needs current-site evidence plus action-specific authorization/gates.
- Follow-up: `source-files-to-site-package.md`, `SKILL.md`, and the new confirmation/execution-plan helpers encode this boundary.

- Context: regression testing for the new confirmation helpers.
- Symptom: the first confirmation test failed because the confirmation did not preserve every package `blockedRemoteActions` item, and the generic sensitive-text scanner treated the ordinary word `authorization` in explanatory fields as credential leakage.
- Evidence: the package builder now includes route/media/settings remote actions such as `bind_route`, `create_route`, `upload_media`, and `save_site_settings`; the confirmation helper copies the package's blocked actions instead of using a smaller static list; the sensitive scanner no longer blocks the ordinary word `authorization` without a credential value.
- Risk: a too-small blocked-action list can silently authorize later route/media/settings mutations, while an overbroad sensitive-word rule can push operators to remove useful boundary language from confirmation artifacts.
- Rule: confirmation validation should be strict about preserving remote action boundaries, but sensitive scanning should target credential-like values, not neutral process words.
- Follow-up: confirmation and execution-plan regression tests cover both boundaries.

- Context: follow-up maintenance after adding source package confirmation and execution planning.
- Symptom: the execution plan sequenced later work, but there was no deterministic export step that turned a confirmed package into runtime upload/planning artifacts. Operators would still need to hand-copy products/posts into manifests and pages/forms/media into separate plans.
- Evidence: `export_confirmed_site_artifacts.py` now writes products/posts draft manifests plus pages, site-info, forms, media, and navigation plan JSON files, and an `artifact-readiness.json` that states `preparedOnly: true`, `remoteMutationsPerformed: false`, and `schemaVerified: false` for draft manifests.
- Risk: hand-copying confirmed content can introduce missing items, duplicate slugs, or accidental schemaVerified claims. It can also blur planning files with replay payloads for pages/forms/media/settings.
- Rule: after source package confirmation, export runtime artifacts once and use those paths as the handoff into schema capture/readiness. Draft manifests should be blocked by `make_manifest_upload_readiness.py` until current-site field mappings and payload templates are captured.
- Follow-up: `SKILL.md` and `source-files-to-site-package.md` now place artifact export between confirmation and manifest/schema validation.

- Context: skill-maintenance round for automating the first raw extraction step after source inventory.
- Symptom: the source workflow still expected a hand-prepared `raw-extraction/summary.json`, so "з”ЁжҲ·з»ҷеҮ дёӘж–Үд»¶иҝҮжқҘ" could not progress deterministically from inventory to source wiki without manual extraction glue.
- Evidence: `extract_source_materials.py` now reads `allincms_source_inventory`, extracts TXT/Markdown/HTML/JSON/CSV/TSV/XLSX/DOCX/PDF text or tables when runtime libraries are available, writes `raw-extraction/extractions.json` plus `summary.json`, and records image files as media candidates needing user approval.
- Risk: without a raw extraction helper, operators can skip inventory-backed extraction, paste large raw source text into the skill, or invent products/posts before the source evidence is visible. Conversely, an extractor can overstep and generate final claims from unreviewed text.
- Rule: raw extraction is evidence preparation only. It may seed a homepage draft and open questions, but final products/posts/site claims must be refined in the source wiki and confirmed by the user before package execution.
- Follow-up: `SKILL.md` and `source-files-to-site-package.md` now route inventory through `extract_source_materials.py` before `build_source_wiki.py`.

- Context: publication-readiness hardening for the same source-file workflow.
- Symptom: structural package validation could pass review-required placeholders such as `Draft Product`, `Draft Article`, `requires review`, or `requires source extraction`, allowing them to reach package confirmation or execution-plan generation even though the user goal is a publishable site.
- Evidence: `validate_source_site_package.py` now supports `--require-publication-ready`; confirmation validation and confirmed execution-plan generation require that stricter gate, while structural validation remains available for dry runs and extraction diagnostics.
- Risk: operators may mistake "source-backed structure exists" for "content is publishable", create a site from placeholder products/posts, or ask the user to confirm content that actually needs extraction/refinement first.
- Rule: before user confirmation or execution planning, run `validate_source_site_package.py --require-complete-package --require-publication-ready`. If it fails, return to the source wiki to replace placeholders, expand thin page/product/post copy, resolve review notes, or record explicit deferrals.
- Follow-up: regression tests cover a publish-ready package passing and a placeholder package failing the strict gate.

- Context: schema-binding hardening after confirmed source-package artifact export.
- Symptom: exported products/posts manifests are intentionally draft manifests, but the workflow still relied on an operator to manually copy `fieldMapping`, `payloadTemplate`, `schemaVerified`, site key, and frontend base URL after save-request capture.
- Evidence: `apply_save_capture_to_manifest.py` now reads a draft posts/products manifest plus validated `allincms_probe_save_capture_evidence`, checks optional base run evidence for site/content-type alignment, applies the captured field mapping and payload template, and requires the output to pass `validate_manifest.py --require-schema-verified`.
- Risk: hand-editing schema flags can accidentally mark the wrong site or content type as upload-ready, mix product and post schemas, or omit the evidence pointer that explains where the payload template came from.
- Rule: after current-site save capture, generate schema-verified manifests with `apply_save_capture_to_manifest.py`. Do not toggle `schemaVerified` or paste payload templates by hand. Products and posts remain separate and still need sample publish/frontend verification before batch upload.
- Follow-up: `SKILL.md`, `source-files-to-site-package.md`, and `batch-verification.md` now route draft manifest upgrades through the helper.

- Context: manifest sample gate hardening between schema verification and batch upload.
- Symptom: the workflow had schema-verified manifests and a batch runbook, but no first-class path for uploading exactly one real manifest item before batch. The existing probe sample path proves a safe test item, not necessarily that the source-generated manifest item maps correctly.
- Evidence: `build_manifest_sample_upload_runbook.py` now prepares a one-slug sample upload/publish runbook, `validate_manifest_sample_upload_evidence.py` validates backend/frontend proof for that slug, and both `build_batch_upload_publish_runbook.py` plus `check_pre_mutation_gate.py` can consume the validated sample evidence before batch upload.
- Risk: operators could jump from schema gate to batch upload, or validate a probe while the actual generated manifest still has body/media/title mapping errors. A second risk is passing sample evidence to the runbook but not to the final pre-mutation gate.
- Rule: for source-generated manifests, upload and validate one manifest sample slug before batch. Pass the same sample evidence to the batch runbook and `check_pre_mutation_gate.py --sample-evidence`; do not rely on probe sample proof alone unless the run explicitly accepts that weaker scope.
- Follow-up: `SKILL.md`, `source-files-to-site-package.md`, and `batch-verification.md` now document the sample evidence bridge.

- Context: local execution-state hardening for long source-file-to-site runs.
- Symptom: the workflow now produces many separate artifacts: package, confirmation, execution plan, artifact readiness, schema manifests, sample evidence, batch evidence, final audit, and launch acceptance. Without one stage summary, an operator can overtrust the most recent passing file and skip the next missing gate.
- Evidence: `summarize_source_execution_status.py` now reads the available local artifacts, reports each stage as passed or blocked, exposes `currentStage`, and emits a concrete `nextAction` without authorizing or performing remote mutations.
- Risk: a run could claim "ready to upload" after schema manifests but before sample proof, or claim "site done" after batch proof but before launch acceptance/forms/media/final audit. Long resumes also become fragile when the next boundary lives only in chat.
- Rule: after each major source-file run stage, refresh `allincms_source_execution_status`. Treat `currentStage` as the next boundary and do not claim full completion until the status is complete and launch acceptance passes.
- Follow-up: `SKILL.md` and `source-files-to-site-package.md` now include the status-summary command.

- Context: review-boundary hardening before source package confirmation.
- Symptom: a publication-ready `allincms_source_site_package` could be confirmed from a chat-only summary, with no deterministic user-review artifact that preserved counts, paths, slugs, source refs, operation gaps, and blocked remote actions.
- Evidence: `make_source_package_review_packet.py` now generates an `allincms_source_package_review_packet` only after the strict publication-ready package gate passes, and `validate_source_package_review_packet.py` binds it back to the package while rejecting count drift, missing blocked actions, sensitive text, overlong previews, and confirmation wording that implies remote mutation authorization.
- Risk: users may approve an incomplete or mis-summarized package, operators may paste long business copy into chat or skill files, or a later run may treat content review as permission to create/save/upload/publish remotely.
- Rule: before package confirmation, generate and validate a review packet. Show or summarize that packet for the user instead of relying on an unstructured chat recap. The packet is local review evidence only; confirmation remains content-intent proof, and remote actions still need action-specific gates.
- Follow-up: `SKILL.md` and `source-files-to-site-package.md` now place the review packet between publication-ready package validation and confirmation record creation.

- Context: follow-up hardening after adding the source package review packet.
- Symptom: even after the review packet existed, `make_source_package_confirmation.py` could still be called by older scripts or habits with only `--package`, allowing confirmation to bypass the stable review artifact. The source execution dashboard also jumped from `source_package` to `confirmation`, so a missing packet could be hidden by a later confirmation file.
- Evidence: `make_source_package_confirmation.py` now requires `--review-packet`; `validate_source_package_confirmation.py` validates `sourceReviewPacket` and can bind confirmation, package, and review packet together; `summarize_source_execution_status.py` now has a separate `review_packet` stage before confirmation.
- Risk: a workflow can regress to chat-only review, stale package summaries, or confirmation artifacts that are not traceable to the review packet the user saw.
- Rule: confirmation must name the exact review packet and package. Status dashboards must block at `review_packet` before `confirmation` when the packet is missing or mismatched.
- Follow-up: `SKILL.md`, `source-files-to-site-package.md`, and regression tests now require review-packet binding.

- Context: confirmed package to new-site creation handoff hardening.
- Symptom: after package confirmation and execution-plan generation, the create-site authorization could still be hand-written from memory. That left a gap where the created site name/description might drift from the confirmed package, or the create-site action could accidentally be bundled with uploads, theme edits, publishing, domains, or tracking.
- Evidence: `build_confirmed_create_site_handoff.py` now validates the publication-ready package, review packet, confirmation, `new_site` execution plan, and fresh create-site preflight together. It outputs suggested authorization text, `make_authorization_record.py` and pre-mutation-gate commands, confirmed siteName/siteDescription, browser steps, and forbidden neighboring actions while keeping `preparedOnly: true` and `isUserAuthorization: false`.
- Risk: content-package approval can be overextended into remote site creation or broader site mutation, or a new site can be created with stale/incorrect name and description that were not the user's confirmed package fields.
- Rule: for source-package-driven new sites, generate a confirmed create-site handoff before asking for create-site authorization. Use it to bind package fields to the `/sites` preflight and to stop after created-site evidence; content upload and publishing remain separate gated stages.
- Follow-up: `SKILL.md`, `source-files-to-site-package.md`, `site-creation.md`, and regression tests now include the handoff stage.

- Context: created-site evidence to upload artifact handoff hardening.
- Symptom: confirmed runtime artifacts can be exported before a new site exists, so products/posts draft manifests may contain placeholders such as `{siteKey-after-creation}` or a stale frontend base URL. If operators hand-edit those fields after site creation, later schema capture and sample upload can point to the wrong site or falsely imply schema readiness.
- Evidence: `bind_created_site_to_artifacts.py` now validates `created_verified` run evidence, reads the real `siteKey` and `frontendBaseUrl`, writes bound products/posts draft manifests plus bound artifact readiness, and keeps `schemaVerified: false`. `summarize_source_execution_status.py` now has a `created_site_binding` stage between artifact export and schema manifests.
- Risk: a new-site run can move from created-site proof to schema/upload planning with placeholder or stale artifact paths, or a manual edit can accidentally preserve schema flags/payload templates from the wrong site.
- Rule: after a new source-package-driven site is created, bind created-site evidence into exported artifacts with the helper. Treat that as site identity binding only; schema capture, sample upload, and frontend proof remain separate gates.
- Follow-up: `SKILL.md`, `source-files-to-site-package.md`, `site-creation.md`, and regression tests now include the created-site artifact binding stage.

- Context: bridge hardening from created-site artifact binding into products/posts schema capture.
- Symptom: bound products/posts draft manifests had the real `siteKey` and `frontendBaseUrl`, but the next step still required operators to hand-plan which content type could create a probe, which content type needed fresh read-only list/edit preflight, and which commands should follow save capture.
- Evidence: `build_schema_capture_handoff.py` now validates the created-site binding, created-site evidence, and bound draft manifests, keeps `schemaVerified: false`, emits per-content-type create-probe authorization templates, and blocks content types whose list/edit preflight is missing or belongs to another content type. `summarize_source_execution_status.py` now has a separate `schema_capture_handoff` stage before `schema_manifests`.
- Risk: operators could treat created-site binding as upload readiness, reuse products field proof for posts, skip read-only preflight, or hand-edit schema flags before live save-request capture.
- Rule: after created-site binding, build the schema-capture handoff. It is local preparation only: no probe creation, no save, no publish, no sample upload, and no batch upload. Products and posts must remain separate until each has its own current-site save capture and schema-verified manifest.
- Follow-up: `SKILL.md`, `source-files-to-site-package.md`, `site-creation.md`, and regression tests now include the schema-capture handoff stage.

- Context: content-type preflight recovery after schema-capture handoff reports that posts or products still need read-only list/edit evidence.
- Symptom: the old setup-page refresh merge preserved created-site proof but did not merge `contentInspection`, so an operator could either keep a content type blocked forever or hand-overwrite created-site evidence with a different content type's fields.
- Evidence: `merge_content_type_preflight.py` now validates created-site evidence, same-site `existing_site_selected` refresh evidence, and the requested content type, then writes a merged evidence file with `contentTypePreflights[contentType]` and top-level `contentInspection` aligned for the normal create-probe gate. Regression tests require the refresh evidence to include the current site key in `existingSiteKeysBeforeCreate`.
- Risk: hand-edited evidence can lose the original create-site proof, unlock a probe on the wrong site, or let products preflight satisfy posts create-probe gates.
- Rule: when schema-capture handoff says a content type needs read-only preflight, generate same-site read-only evidence for that content type, merge it with `merge_content_type_preflight.py`, and rebuild the handoff. Do not edit `contentInspection` by hand or use refresh evidence whose selected site list does not include the current site key.
- Follow-up: `SKILL.md`, `source-files-to-site-package.md`, `site-creation.md`, and schema-capture tests now include the preflight merge recovery path.

- Context: request-analysis turn where the user asked whether the current skill can help with batch upload.
- Symptom: the skill had source-package, schema-binding, manifest-sample, and batch-runbook helpers, but the most important answer was conditional: draft/source manifests are not upload-ready until live schema and sample gates pass.
- Evidence: `make_manifest_upload_readiness.py`, `build_manifest_sample_upload_runbook.py`, and `build_batch_upload_publish_runbook.py` enforce separate schema, sample, authorization, and pre-mutation gates before `batch_upload`.
- Risk: a future operator can answer "yes, it supports batch upload" too broadly and then batch from source files, source-site packages, exported draft manifests, or `schemaVerified: false` manifests.
- Rule: when asked whether batch upload is available, answer with the current gate state: supported after schema-verified manifest, validated save-capture evidence, one manifest sample backend/frontend proof, action-specific batch authorization, and `check_pre_mutation_gate.py --action batch_upload`. Anything earlier is preparation only.
- Follow-up: `SKILL.md` now states the batch-upload capability boundary near the top-level operating rule.

- Context: local schema-capture queue hardening during the same batch-upload readiness review.
- Symptom: after `build_schema_capture_handoff.py` reported a content type ready for create-probe authorization, the workflow still had no compact status file showing whether create-probe evidence, save handoff, save runbook, save-capture evidence, and schema manifest were actually present for that content type.
- Evidence: `summarize_schema_capture_progress.py` now reads the schema-capture handoff plus optional per-content-type artifacts and reports states such as `ready_for_create_probe`, `ready_for_save_handoff`, `ready_for_save_capture`, `ready_to_apply_save_capture`, and `schema_manifest_ready`. It validates save handoff/runbook kind and content type before advancing.
- Risk: operators could skip from create-probe readiness directly to manifest sample or batch upload, especially after resuming from scattered `/tmp` artifacts.
- Rule: after schema-capture handoff, refresh the schema-capture progress summary and follow `results[].status` per content type. `schema_manifest_ready` unlocks only the manifest sample runbook; batch upload still needs sample backend/frontend proof and a batch mutation gate.
- Follow-up: `SKILL.md` and `source-files-to-site-package.md` now route schema-capture resumes through `summarize_schema_capture_progress.py`.

- Context: goal-continuation turn for user-file-to-wiki-to-site-package flow.
- Symptom: the source-file workflow had deterministic helpers for inventory, extraction, wiki, requirements, package, review packet, and status, but a user handing over several files still forced the operator to manually chain many commands before knowing whether the package was ready for review.
- Evidence: `prepare_source_site_package.py` now runs the local preparation chain in one command, writes the standard runtime artifacts, and reports `packageStatus` as `review_ready`, `needs_source_wiki_refinement`, or `blocked_structural_package`.
- Risk: without a total local preparation entrypoint, operators can skip raw inventory, miss requirements/gaps, or jump from a thin extracted summary to remote site creation. Conversely, an orchestrator could falsely imply that deterministic extraction is the same as AI-quality publication copy.
- Rule: start user-file runs with `prepare_source_site_package.py`, then follow its `packageStatus`. Only `review_ready` may produce a review packet for user content-intent confirmation. `needs_source_wiki_refinement` means the AI/operator must refine the source wiki/package from the extracted source material before confirmation, site creation, schema capture, or upload planning.
- Follow-up: `SKILL.md`, `source-files-to-site-package.md`, and regression tests now include the local preparation orchestrator.

- Context: confirmed-package execution preparation after user content-intent confirmation.
- Symptom: after a user confirmed a review packet, the workflow still required hand-chaining confirmation record creation, execution-plan generation, artifact export, execution status, and optionally create-site handoff generation.
- Evidence: `prepare_confirmed_site_execution.py` now generates the confirmation record, confirmed execution plan, exported draft artifacts, source execution status, and, when a fresh create-site preflight is supplied, the confirmed create-site handoff with authorization placeholders.
- Risk: a user confirmation could be overextended into remote site creation or upload authorization, or operators could skip artifact export and try to upload the source package directly.
- Rule: after review-packet confirmation, run `prepare_confirmed_site_execution.py` and follow `readyForBrowserStage`. `create_site_handoff_ready` means safe to request action-time create-site authorization; `needs_create_site_preflight` means read-only `/sites` preflight is still missing; `ready_for_existing_site_readonly_refresh` means selected-site evidence must be refreshed before binding artifacts or schema capture.
- Follow-up: `SKILL.md`, `source-files-to-site-package.md`, and regression tests now include the confirmed execution preparation orchestrator.

- Context: created-site evidence to schema-capture preparation after a confirmed new-site run.
- Symptom: once a site was created and verified, the workflow still required manually binding `siteKey`/frontend base URL into exported draft manifests, then separately building schema-capture handoff and progress summaries.
- Evidence: `prepare_created_site_schema_capture.py` now runs created-site artifact binding, schema-capture handoff generation, schema-capture progress summary, and source execution status refresh in one local-only step.
- Risk: operators could hand-edit manifests, accidentally mark `schemaVerified: true`, skip a content type's read-only preflight, or jump from created-site evidence directly to upload planning.
- Rule: after `created_verified` evidence, run `prepare_created_site_schema_capture.py`. Follow `readyForCreateProbeAuthorizationCount` and `blockedByReadonlyPreflightCount`; bound manifests must remain `schemaVerified: false`, and create-probe remains a separate action-time authorization stage.
- Follow-up: `SKILL.md`, `source-files-to-site-package.md`, and regression tests now include the created-site schema-capture preparation orchestrator.

- Context: integrating pages/site-info handoff into the created-site continuation orchestrator.
- Symptom: `prepare_pages_site_info_execution.py` existed, but the normal new-site continuation path still required a separate manual call after created-site evidence. That could leave page/site-info execution out of the source execution status while products/posts schema capture advanced.
- Evidence: `prepare_created_site_schema_capture.py` now reads `pagesPlan` and `siteInfoPlan` from artifact readiness, builds `pages-site-info/pages-site-info-browser-handoff.json`, and passes it into `summarize_source_execution_status.py`.
- Risk: a user-file-to-site run can appear to have a coherent next action for content upload while homepage/single-page/site-info actions were never prepared from the confirmed package.
- Rule: in the normal new-site path, use `prepare_created_site_schema_capture.py` after `created_verified` evidence; it prepares both pages/site-info and products/posts schema-capture handoffs. Use `prepare_pages_site_info_execution.py` directly only for existing-site continuation or one-stage rebuilds.
- Follow-up: `SKILL.md`, `source-files-to-site-package.md`, and created-site preparation tests now reflect the integrated handoff.

- Context: create-probe evidence to save-capture preparation in the schema-capture lifecycle.
- Symptom: after a content type's create-probe stage produced redacted evidence and an edit URL, operators still had to hand-chain `prepare_probe_save_handoff.py`, `build_probe_save_runbook.py`, and `summarize_schema_capture_progress.py`.
- Evidence: `prepare_schema_save_capture.py` now builds the save handoff, save runbook, and refreshed schema-capture progress for one content type from schema-capture handoff plus create evidence.
- Risk: create-probe authorization could be overextended into save/publish/cleanup, or operators could hand-write save authorization text/runbooks with the wrong content type or preflight evidence.
- Rule: after create-probe evidence, run `prepare_schema_save_capture.py` for that exact content type. `progressStatus: ready_for_save_capture` means safe to request action-time `save_probe` authorization; it does not save, capture schema, publish, cleanup, sample upload, or batch upload.
- Follow-up: `SKILL.md`, `source-files-to-site-package.md`, and regression tests now include the schema save-capture preparation orchestrator.

- Context: validated save-capture evidence to schema-verified manifest and sample-runbook preparation.
