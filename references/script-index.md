# 脚本索引

`scripts/` 下辅助脚本的用途索引(每条脚本自己也有 `--help`)。按用途查"哪个脚本做什么";
实际手工调用的完整命令在 `SKILL.md` 的 Workflow 各编号步骤里。运行时/门禁脚本、校验器、
handoff/brief 构造器都在此列全。

- `scripts/make_create_preflight_evidence.py`: generate a pure read-only create-site preflight evidence JSON from observed site keys before submitting the create form.
- `scripts/make_create_preflight_from_existing_site_evidence.py`: derive a `create_preflight_verified` create-site preflight file from fresh `existing_site_selected` read-only evidence after a real `/sites` refresh. Use it when the browser refresh already captured site keys and scoped create-dialog fields, but the next stage is `create_site_submit`.
- `scripts/make_existing_site_readonly_evidence.py`: generate read-only evidence after selecting and inspecting an existing site without submitting, saving, uploading, publishing, or deleting.
- `scripts/redact_browser_scan.py`: redact raw browser scan JSON before storage, scan-to-evidence conversion, or handoff; removes account/menu noise, emails, frontend domain labels, and long raw text while preserving backend module URLs and neutral controls.
- `scripts/make_existing_site_evidence_from_scan.py`: convert a redacted browser scan JSON into existing-site read-only evidence to avoid hand-copying long refresh commands. It accepts protected module URLs with `{siteKey}` or `{realSiteKey}` placeholders after redaction and reconstructs concrete module routes from the verified `siteKey`. For non-table setup/grid pages such as media, site-info, tracking, and domains, it can use visible controls/inputs as the content inspection proof instead of requiring table headers.
- `scripts/make_created_site_evidence.py`: upgrade preflight evidence after a real create-site submit and current new-site verification; optionally merge generated frontend rendering and launch readiness evidence into the same run-evidence file.
- `scripts/merge_created_site_readonly_refresh.py`: merge current read-only setup-page evidence into older `created_verified` site evidence when validators add required setup modules or a fresh read-only refresh fills missing setup keys. It preserves the original create timestamp and authorization, so use it only as phase evidence; it does not make stale evidence fresh for mutation.
- `scripts/make_authorization_record.py`: generate an action-specific mutation authorization record; use before create-site submit, draft creation, existing product/post save or publish, probe save/publish/cleanup, upload, cleanup, or batch operations.
- `scripts/check_pre_mutation_gate.py`: verify that preflight/read-only evidence and action-specific authorization match before create-site submit, content-type-specific probe creation, existing product/post save or publish, batch operations, media upload, and site/theme/form/domain/tracking actions.
- `scripts/simulate_full_e2e_chain.py`: run the full local-only dry run by orchestrating site creation, static launch evidence, module scan summary, staged capture plan, content probe lifecycle, summaries, and closeout in one command.
- `scripts/simulate_site_creation_chain.py`: run a local-only dry run of preflight, authorization, final gate, created-site evidence, run summary, closeout gate, and validation before real browser submit.
- `scripts/simulate_probe_lifecycle.py`: run a local-only staged dry run from created/read-only evidence through probe creation, request capture, sample verification, cleanup, final summary, and closeout.
- `scripts/validate_manifest.py`: run before posts/products batch upload when the source content is represented as JSON. Use `--json` for machine-readable validation in rehearsals or parallel checks. Media, themes, routes, and forms need their own schema-specific validator after request capture.
- `scripts/validate_slate_content_shape.py`: run on a products/posts manifest before it enters the batch to assert every `content` field is a Slate node array (`[{type,children:[{text}],id?}]`), not a markdown/HTML string. A UI form save does not bind Slate, so a string body persists empty on the live frontend; this gate catches that silent-empty-body failure locally. Use it right after the AI content pass and before sample/batch upload.
- `scripts/check_next_action_freshness.py`: run right before a JSON batch to refuse replay when the captured save contract's `deploymentId` does not match the current live deployment, when the deployment marker is missing, or when required next-action ids are absent. `next-action` ids change per deployment and a stale id fails silently, so read the current deployment id live and pass `--current-deployment-id` before the batch.
- `scripts/check_content_quality.py`: pre-publish CONTENT-quality gate (complements the structural `validate_source_site_package.py`). Scans `contentPlan.products[*]`/`posts[*]` and BLOCKS on stock/placeholder images (in `coverImage`/`media` object-or-array/`gallery`/top-level `contentPlan.media`/content body), missing alt on a main product image, and unreplaced placeholder/template markers; WARNS on off-category/leftover-demo terms, a thin spec table (<3), duplicate copy across items, gallery images without alt, and `example.com` references; and returns an `aiReviewChecklist` for judgments that can't be automated (hallucinated specs, unbacked differentiators, fabricated trust claims). `--strict` promotes warnings to blockers. Run before a content batch/publish.
- `scripts/validate_theme_page_document.py`: validate a theme design-save JSON replay body (`[{siteId,themeId,pageId,intent:'save',pageDocument:{root,elements:{id:{type,props}}}}]`) before sending it. Use `--require-publication-ready` to also reject placeholder copy and local/relative image paths in block props. This is the local half of theme-JSON replay: once a live CDP capture supplies the current design-save `next-action` id, the whole-page replay body is validated by this gate (see `references/server-action-save-api.md` §7).
- `scripts/make_source_input_requirements.py`: generate a run-evidence JSON that lists which user/PDF/catalog/website/spreadsheet fields are required, recommended, source-derived, user-confirmed, or blocked until schema capture. Use it before extracting source materials into posts/products/forms/media/theme-page/site-info/routes/domains/tracking/navigation manifests or runbooks.
- `scripts/record_source_input_gap.py`: append per-operation field gaps into a local run ledger outside the skill package. Use it while operating the browser whenever a field is discovered that later needs user input, PDF/catalog extraction, schema capture, media proof, or an explicit omit/acceptance decision. Feed that ledger back into `make_source_input_requirements.py --gap-ledger` before extracting user PDFs/catalogs/websites/spreadsheets so the source-intake contract includes browser-discovered gaps.
  Run it again with `--validate-only` before using an older or merged ledger; this checks that every gap row still has source hints, generation rules, evidence pointers, proof notes when needed, no sensitive strings, no duplicates, and a matching summary.
- `scripts/make_resolved_source_input_gaps.py`: create a separate `allincms_resolved_source_input_gaps` evidence file when later browser proof supersedes an older operation gap. Use this instead of editing or deleting old ledger rows. Pass the generated file to `make_source_input_requirements.py --resolved-gap-evidence`, `prepare_source_site_package.py --resolved-gap-evidence`, or `run_source_file_rehearsal.py --resolved-gap-evidence` so stale blockers are filtered while the original gap audit trail remains intact.
- `scripts/make_manifest_upload_readiness.py`: build a multi-manifest readiness report after draft validation. Use it when products and posts are prepared together so `blocked` schema gates are visible and generic validation is not mistaken for upload readiness.
- `scripts/build_source_inventory.py`: create a local source inventory from user files/directories before raw extraction, wiki distillation, or package generation.
- `scripts/extract_source_materials.py`: extract inventory files into local raw-extraction artifacts and a conservative extraction summary for the source-wiki builder. Use it for PDFs, DOCX, spreadsheets, CSV/TSV, Markdown, TXT, JSON, HTML, and image metadata.
- `scripts/build_source_wiki.py`: build a local `allincms_source_wiki` JSON from a source inventory and optional extraction summary.
- `scripts/validate_source_wiki.py`: validate the source wiki against the inventory, source refs, required pages/products/posts/site fields, and leakage/raw-Markdown checks.
- `scripts/export_source_wiki_markdown.py`: export `allincms_source_wiki` into local readable Markdown wiki files (`index.md`, `site.md`, `pages.md`, `products.md`, `posts.md`) for user/operator review. These files are review artifacts, not upload payloads.
- `scripts/upload_media_via_picgo.py`: upload the local source images referenced by an `allincms_source_wiki` (root `media[]`, per-item `mediaNeeds`, and Markdown `![alt](path)` refs in body content) through a PicGo local server (default `http://127.0.0.1:36677/upload`), then rewrite every local image link to the returned hosted URL and write an `allincms_media_upload_map` evidence file. This is not an AllinCMS mutation (`allincmsRemoteMutationsPerformed=false`), but it performs a real external upload that makes images public, so it defaults to `--dry-run` (plan only) and requires `--confirm-upload` for a real upload — run the real upload only after the user has approved the media in content confirmation (`mediaPolicy.requiresUserApprovalBeforeUpload`). Never store image bytes, PicGo config, or tokens in the skill package.
- `scripts/make_source_wiki_refinement_plan.py`: convert source-wiki/package/review validation issues into field-level source-wiki repair actions. Use it when source packages are not review-ready so the next AI pass knows which page/product/post/site-info/media/contact/taxonomy fields to rewrite, source, confirm, or defer.
- `scripts/make_source_wiki_refinement_brief.py`: turn the source wiki, refinement plan, and optional Markdown wiki exports into a safe AI/operator brief for rewriting `allincms_source_wiki`. Use it after `needs_source_wiki_refinement` so the next pass has explicit targets, output contract, validation command, and remote-mutation prohibitions.
- `scripts/draft_refined_source_wiki.py`: generate a local refined source-wiki draft from the current source wiki and refinement brief when extracted products/posts are usable but pages/site-info/navigation/policies need deterministic expansion before review. It can expand thin pages, add homepage/navigation/policy defaults, and validate the same output contract; it does not replace user confirmation or live schema capture.
- `scripts/prepare_source_site_package.py`: run the local source-file preparation chain in one command: inventory, raw extraction, source wiki JSON, readable Markdown wiki, source-input requirements, source-site package, optional review packet, and source execution status. It never performs remote mutation; if the output is not `review_ready`, refine the source wiki/package before asking for confirmation.
- `scripts/apply_refined_source_wiki.py`: after AI/user refinement turns `source-wiki.json` into a publication-ready `allincms_source_wiki`, export readable Markdown wiki files, rebuild and validate the source-site package, generate the review packet, and refresh source execution status in one local-only step. Use it to move `needs_source_wiki_refinement` runs toward user confirmation without hand-wiring package/review/status commands.
- `scripts/validate_refined_source_wiki_contract.py`: validate a refined `allincms_source_wiki` against the refinement brief that requested it. Use it after AI rewrites source wiki content or before applying a copied refined wiki; it checks output path binding, source refs, non-authorization flags, placeholder removal, navigation/media/contact/taxonomy presence, and optional inventory validation.
- `scripts/run_source_file_rehearsal.py`: local-only orchestration for real "user gave files" runs. It calls initial source preparation, can auto-draft a refined source wiki with `--auto-draft-refined-source-wiki`, optionally applies a refined source wiki, and optionally prepares confirmed execution artifacts when user confirmation text is supplied. Use it to prove whether the run is blocked at source-wiki refinement, ready for content confirmation, or ready for browser create/select-site work. It does not create sites, save content, upload, publish, or authorize mutations.
- `scripts/validate_source_file_rehearsal.py`: validate `source-file-rehearsal-summary.json` before showing a confirmation brief, reusing an older rehearsal, or continuing to browser work. It reopens package/review/brief artifacts, checks objective-audit and stage drift, and rejects sensitive raw header/account text.
- `scripts/make_source_confirmation_brief.py`: convert `source-file-rehearsal-summary.json` into compact `source-confirmation-brief.json` and optional Markdown for user-facing content confirmation. It merges `confirmationReview` and `objectiveAudit`, exposes counts, coverage, suggested confirmation text, deferrals, command templates, and the next blocker, while keeping `isRemoteMutationAuthorization=false`.
- `scripts/validate_source_confirmation_brief.py`: validate the confirmation brief against its source rehearsal summary. Use it before showing or reusing a copied brief; it rejects authorization leakage, missing command templates, coverage/count drift, objective completion overclaiming, and summary mismatch.
- `scripts/prepare_source_confirmation_next_step.py`: consume a validated `source-confirmation-brief.json` and emit the next local handoff from its `executionIntake`. Use it after the user confirms content intent or when resuming from a brief: it can prepare the local confirmed-execution command, identify the read-only create-preflight target, or point to the non-executable gated create-site runbook/evidence bundle. It never authorizes or performs remote mutation.
- `scripts/apply_source_confirmation_next_step.py`: apply a `source-confirmation-next-step` handoff only for local stages. Use it when the handoff has `localCommandReady=true` to generate confirmed execution artifacts without parsing shell commands by hand. It intentionally stops at browser boundaries such as read-only create preflight or gated create-site runbook, and never authorizes or performs remote mutation.
- `scripts/apply_create_preflight_to_confirmed_execution.py`: after read-only `/sites` create preflight evidence is collected, consume the prior source-confirmation apply result plus the filled `create-site-preflight.json` and regenerate confirmed execution artifacts with create-site handoff, validation, non-executable runbook, and created-site evidence bundle. Use it to avoid reconstructing the long confirmed-execution command from confirmation text and deferrals. It remains local-only and does not authorize or submit the create-site form.
- `scripts/build_source_site_package.py`: build a local `allincms_source_site_package` JSON from a distilled source wiki. Use it before user confirmation and before any remote create/upload stage.
- `scripts/validate_source_site_package.py`: validate the local source-site package and enforce no remote mutation, source refs, site-info/navigation presence, confirmation gate, draft schema state, and no raw Markdown/credential leakage.
- `scripts/make_source_package_review_packet.py`: create a redacted local review packet from a publication-ready source-site package before asking the user to confirm. It summarizes site info, navigation keys, page/product/post counts, slugs, paths, source refs, operation gaps, blocked actions, and suggested confirmation text without storing long body copy or raw source text.
- `scripts/validate_source_package_review_packet.py`: validate the review packet against the source package and reject sensitive text, full-copy leakage, count drift, missing blocked actions, or language that implies remote mutation authorization.
- `scripts/make_source_review_objective_coverage.py`: summarize a validated source-package review packet into the pre-browser objective checklist. Use it before asking for content-intent confirmation or answering whether source files are ready for site creation/upload. It can return `reviewComplete=true`, but the full objective still has `complete=false` until live site creation/selection, schema capture, sample upload, batch upload, frontend launch, cleanup, and closeout are proven.
- `scripts/make_source_package_confirmation.py`: create the local user-confirmation record for a validated source-site package after the review packet has been shown or summarized. It requires `--review-packet`, confirms content intent only, and must not be treated as remote mutation authorization.
- `scripts/validate_source_package_confirmation.py`: validate the confirmation record against the source package and review packet, including accepted site info/navigation/content fields, confirmed counts, blocked remote actions, review-packet binding, and local-only/no-mutation flags.
- `scripts/prepare_confirmed_site_execution.py`: after the user confirms a review packet, prepare the local execution bundle in one step: confirmation record, execution plan, exported draft artifacts, source execution status, create-site preflight brief when preflight is missing, and create-site handoff plus non-executable browser runbook when fresh create-site preflight is provided. It remains local-only and does not authorize remote actions.
- `scripts/build_confirmed_site_execution_plan.py`: after package confirmation, build a prepared-only execution plan that sequences create/select site, setup inspection, pages/site-info handoff and execution, schema capture, sample upload, batch upload, forms/media/settings, final audit, and cleanup while preserving action-specific authorization boundaries.
- `scripts/make_create_site_preflight_brief.py`: after package confirmation and a `new_site` execution plan, build a non-authorizing read-only browser brief for collecting `/sites` and create-dialog evidence needed by `make_create_preflight_evidence.py`. Use it when confirmed execution reports `needs_create_site_preflight`.
- `scripts/make_created_site_evidence_brief.py`: after a confirmed create-site handoff is ready, build a non-authorizing post-submit evidence brief for collecting the new site key, backend/frontend URLs, module routes, setup-page proof, and one content-type read-only inspection required by `make_created_site_evidence.py`.
- `scripts/build_confirmed_create_site_handoff.py`: after package confirmation, execution-plan creation, and fresh create-site preflight, build a non-authorizing handoff for creating the new site from confirmed package siteName/siteDescription. It prepares suggested authorization text, authorization-record command, pre-mutation gate command, browser steps, and forbidden neighboring actions.
- `scripts/build_create_site_runbook.py`: turn a confirmed create-site handoff into a concrete browser runbook for opening `/sites`, filling the create dialog, submitting once after action-time authorization/gate, and writing created-site evidence. It keeps `browserStepsExecutable=false` and must be followed by real browser evidence, not treated as proof that a site was created.
- `scripts/prepare_create_site_authorization.py`: consume a create-site apply result or runbook and prepare the action-time authorization boundary. Without current user authorization text it writes a non-authorizing package with suggested authorization text and command templates. With exact current user authorization text it writes the local authorization record and validates `check_pre_mutation_gate.py` for `create_site`, while still not submitting the form or making the runbook executable.
- `scripts/prepare_created_site_evidence_bundle.py`: after the create-site runbook and created-site evidence brief exist, prepare a local evidence bundle with a fillable redacted evidence template, notes, an apply command, the lower-level `make_created_site_evidence.py` command file, and `prepare_created_site_schema_capture.py` command file. The template must later record the authorization record and `preMutationGateStatus=passed`; the bundle itself is scaffolding only and does not authorize or prove site creation.
- `scripts/apply_created_site_evidence_bundle.py`: after the gated create-site browser submit, consume the filled created-site evidence bundle template, require a concrete authorization record and passed pre-mutation gate proof, write standard `created-site-evidence.json`, and optionally run `prepare_created_site_schema_capture.py` in the same local-only apply step. Use this instead of hand-copying many template fields into `make_created_site_evidence.py`.
- `scripts/apply_created_site_evidence_to_source_rehearsal.py`: after `apply_create_preflight_to_source_rehearsal.py` and one gated browser create-site submit produced a filled created-site evidence template, consume the source apply result and filled template, reuse the created-site evidence bundle, reload package/review context from confirmation, apply created-site evidence, and prepare schema/pages/site-info/taxonomy artifacts in one local-only step. Use it instead of hand-stitching package, review packet, confirmation, execution plan, artifact readiness, and evidence bundle paths after a source rehearsal.
- `scripts/export_confirmed_site_artifacts.py`: export a confirmed source package into runtime artifacts: products/posts draft manifests, pages/site-info/forms/media/navigation plans, and artifact readiness. Use it after confirmation and before schema capture/upload planning.
- `scripts/prepare_pages_site_info_execution.py`: after confirmed artifacts exist and the created/selected site has current setup-page preflight evidence, convert `pages-plan.json` and `site-info-plan.json` into a local browser handoff for site-info save and page/theme/route actions. It keeps every action non-executable until concrete targets, action-time authorization, and pre-mutation gates exist.
- `scripts/make_theme_page_target_map.py`: after read-only theme page list/design URL inspection, convert `pages-site-info-browser-handoff.json` plus a redacted observation JSON into a concrete theme/page target map. Use it before authorizing `save_design`, `publish_design`, `enable_theme_page`, or `bind_route` so `{themeId}` and `{pageId}` placeholders are replaced by current-site URLs, and missing source pages are separated for `create_theme_page`.
- `scripts/prepare_pages_site_info_evidence_bundle.py`: after a pages/site-info browser handoff exists, prepare a local evidence bundle with a fillable execution-evidence template, notes, validation command, and apply command. It is scaffolding only; browser actions still need per-action authorization/gates and redacted backend/frontend proof.
- `scripts/prepare_default_theme_bootstrap.py`: when a newly created or selected site's public frontend is 404/blank and themes are missing or have zero usable pages, prepare a local default-theme bootstrap runbook. It generates separate `create_theme` and `activate_theme` authorization commands, requires preset `默认`, and writes a redacted evidence template. It does not create or activate a theme.
- `scripts/validate_default_theme_bootstrap_evidence.py`: validate redacted evidence after a default-theme bootstrap run. It requires preset `默认`, concrete `themeId`, generated starter page count, theme activation proof, route checks, public non-empty frontend DOM for required starter paths, and no raw credential/header material. Passing this fixes the blank-theme foundation only; business content, homepage modules, products, posts, forms, media, and launch QA remain separate gates.
- `scripts/apply_default_theme_bootstrap.py`: after default-theme bootstrap evidence validates, apply that foundation proof back into created-site evidence and write `created-site-evidence.after-default-theme-bootstrap.json`. Use the refreshed evidence for later pages/site-info, taxonomy, and schema-capture preparation. When `--prepare-created-site-schema-capture` plus artifact-readiness context is supplied, it also reruns the local created-site preparation from the refreshed evidence and emits the downstream source status/next-stage handoff. It does not create, activate, save, publish, upload, or claim business content complete.
- `scripts/prepare_taxonomy_execution.py`: after `taxonomy-plan.json` exists and the created/selected site has current products/posts read-only evidence, prepare non-executable category/tag create-or-map actions for product and post taxonomy. It does not create or map terms; each term action still needs current UI/request schema, action-time authorization, and pre-mutation gate. Category create is a capturable Server Action that returns a `categoryId` — prefer JSON replay over UI clicking (see `references/server-action-save-api.md` §2), and run taxonomy first: products/posts reference categories as an ID array, so create categories and record each `categoryId` before any product/post payload references them.
- `scripts/validate_taxonomy_execution_evidence.py`: validate redacted browser evidence after taxonomy terms were created or mapped. Use it before relying on categories/tags in products/posts manifests or homepage category modules.
- `scripts/prepare_taxonomy_evidence_bundle.py`: after a taxonomy execution handoff exists, prepare a local evidence bundle with a fillable taxonomy evidence template, notes, validation command, and apply command. It is scaffolding only; browser actions still need per-term authorization/gates and redacted backend mapping proof.
- `scripts/apply_taxonomy_execution.py`: after authorized taxonomy browser work has produced redacted create/map evidence, validate it and refresh source execution status in one local-only step. Use it to advance from `taxonomy_execution` to schema or manifest gates without hand-wiring status arguments.
- `scripts/bind_created_site_to_artifacts.py`: after a new site has `created_verified` evidence, or an existing site has fresh `existing_site_selected` read-only evidence, bind the real `siteKey` and `frontendBaseUrl` into exported draft products/posts manifests and artifact readiness while keeping `schemaVerified: false`. Existing-site binding is continuation proof only, not proof that a new site was created.
- `scripts/prepare_created_site_schema_capture.py`: after created or selected-site evidence exists, run site artifact binding, pages/site-info browser handoff preparation, taxonomy execution handoff preparation, schema-capture handoff, schema-capture progress, and source execution status in one local-only step. It does not create probes, create/map taxonomy, save site-info, create pages, save design, publish, upload, or verify schemas.
- `scripts/apply_pages_site_info_execution.py`: after authorized pages/site-info browser work has produced redacted execution evidence, validate it and refresh source execution status in one local-only step. Use it to advance from `pages_site_info_execution` to taxonomy or schema gates without hand-wiring status arguments.
- `scripts/build_schema_capture_handoff.py`: after created/selected-site artifact binding, prepare the per-content-type schema-capture handoff for bound products/posts draft manifests. It creates no probe and performs no save; it only reports which content types are ready for create-probe authorization and which still need read-only list/edit preflight.
- `scripts/merge_content_type_preflight.py`: after a read-only products/posts refresh, merge that content type's list/edit evidence into created-site evidence so `build_schema_capture_handoff.py` and the create-probe gate can use the correct per-content-type preflight without losing the original site creation proof.
- `scripts/summarize_schema_capture_progress.py`: after schema-capture handoff, create-probe evidence, save handoff/runbook, save-capture evidence, and schema manifests start accumulating, summarize the exact per-content-type state. It reads local artifacts only and prevents jumping from create-probe readiness directly to manifest sample or batch upload.
- `scripts/apply_save_capture_to_manifest.py`: after one content-type save request capture is validated, apply its `fieldMapping` and `payloadTemplate` to that content type's draft manifest and write a schema-verified manifest. This is local schema binding only; it does not upload, publish, or prove sample/frontend success.
- `scripts/prepare_schema_manifest_sample.py`: after validated save-capture evidence exists, generate the schema-verified manifest, manifest upload readiness report, one-item sample upload runbook, optional schema-capture progress, and optional refreshed source execution status. It does not upload or publish the sample.
- `scripts/build_manifest_sample_upload_runbook.py`: after a schema-verified manifest exists, prepare the one-item sample upload/publish runbook for a selected manifest slug. It remains non-executable until action-time authorization and the pre-mutation gate pass.
- `scripts/prepare_manifest_sample_evidence_bundle.py`: after a manifest sample runbook exists, prepare a local evidence bundle with a fillable sample evidence template, notes, validation command, and apply command. It is scaffolding only; browser actions still need sample authorization/gate and redacted backend/frontend proof.
- `scripts/validate_manifest_sample_upload_evidence.py`: validate redacted evidence from one manifest sample upload/publish and frontend detail check. If the sampled manifest item contains `coverImage`, `media`, or `gallery`, the sample evidence must prove `coverOrMediaVerified=true`; note-only absence is allowed only for no-media manifest items. Batch upload remains blocked until this evidence passes for the same manifest/site/content type.
- `scripts/apply_manifest_sample_upload.py`: after one authorized source-generated manifest sample upload has produced redacted evidence, validate it, write a progress entry, and refresh source execution status in one local-only step. Use it to advance from `sample_upload` to `batch_upload` without hand-wiring status arguments.
- `scripts/prepare_batch_upload_publish.py`: after validated manifest sample evidence exists, validate the sample against the schema-verified manifest, seed the batch progress log with `mediaRequired` flags per slug, and build the batch upload/publish runbook plus evidence bundle in one local-only step. It does not upload, publish, or authorize browser execution.
- `scripts/prepare_batch_upload_publish_evidence_bundle.py`: after a batch upload/publish runbook exists, prepare a local evidence bundle with a fillable batch evidence template, progress log target, final-audit commands, validation command, and apply command. It is scaffolding only; browser actions still need batch authorization/gate and redacted per-slug backend/frontend proof.
- `scripts/apply_batch_upload_publish.py`: after an authorized batch upload/publish run has produced redacted evidence, validate it, write a batch progress log, and refresh source execution status in one local-only step. Use it to advance from `batch_upload` to `forms_media_settings`; launch acceptance still needs forms/media/settings proof or explicit deferrals.
- `scripts/validate_forms_media_settings_evidence.py`: validate redacted forms/media/settings proof or explicit deferrals after the browser stage. It rejects missing per-module decisions and raw credentials/request material.
- `scripts/prepare_forms_media_settings_evidence_bundle.py`: when source status is at `forms_media_settings`, generate a local fillable evidence bundle (`forms-media-settings-evidence.filled.json`, notes, validation command, apply command) before browser proof. Use the bundle path with `prepare_source_next_stage.py --forms-media-settings-evidence-bundle` after the filled proof exists.
- `scripts/apply_forms_media_settings.py`: after forms/media/settings proof or explicit deferrals exist, validate the evidence, write `forms-media-settings-validation.json`, and refresh source execution status in one local-only step. Use it to advance from `forms_media_settings` to `launch_acceptance`; final audit, cleanup, and launch acceptance still remain separate gates.
- `scripts/prepare_launch_acceptance_inputs_bundle.py`: when source status is at `launch_acceptance`, generate a local fillable inputs bundle (`launch-acceptance-inputs.filled.json`, notes, validation command, apply command) for final run evidence, module coverage, final frontend audit, cleanup, closeout, and source-context paths. Use it with `prepare_source_next_stage.py --launch-acceptance-inputs-bundle` after final browser proof exists.
