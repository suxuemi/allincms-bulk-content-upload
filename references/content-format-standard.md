# 正文格式与信息架构规范(产品 / 文章正文怎么排成专业级)

`source-material-norms.md` 定义「正文该供什么内容」(what / 差异点 / applications);本文定义
「正文该怎么**排版**」——标题层次、加粗、列表、循序渐进的信息架构、结尾 CTA。两者缺一,就会
出现「内容对题却是段落墙」的页面(规格埋在散文里、读者扫不到重点、走不到下一步)。

## 第一原则:正文是给**扫读者**做决策,不是段落墙

B2B/电商访问者不逐字读——他 5 秒扫**标题 + 加粗 + 列表**,判断「是什么、强在哪、怎么买」。
所以正文必须**可扫读**:分节标题引路、关键处加粗、要点用列表、每段短、结尾一个明确的下一步。
一整屏连续段落 = 读者划走。

## 产品正文信息架构(H2 分节,按访问者决策旅程)

| # | 分节(H2) | 放什么 | 格式 |
|---|---|---|---|
| 1 | (开篇,无标题) | 1 句「是什么 + 给谁 + 头号价值」 | 关键规格/价值**加粗**,2-3 句 |
| 2 | **关键规格 / 选型参数** | 规格表 + 一句点出决定选型的头 2-3 个参数 | 关键值**加粗** |
| 3 | **为什么选它 / 核心优势** | ≥2 个差异点,每个绑数字或机制 | **bullet 列表**,卖点词加粗 |
| 4 | **典型应用** | 不同场景 / 客户类型 | **bullet 列表** |
| 5 | **品质与信任**(有才写) | 材料/工艺/标准/认证/质保/测试数据 | 可证实陈述,不编造 |
| 6 | **获取报价 / 下一步(CTA)** | 怎么配、找谁询、索取 datasheet | 明确动作,正文**必有** |

文章正文同理(H2 分节):痛点 hook →(H2)机制/权衡 →(H2)按名映射产品 →(H2 或结尾)payoff + 软 CTA。

## 格式硬规则

- **标题层次**:正文 ≥2 个 H2 分节;别整屏段落。长节可用 H3 细分。
- **加粗**:关键规格值、核心卖点、CTA 动作**加粗突出**——但**克制**,每段 ≤1-2 处,不满屏加粗(满屏加粗=没有重点)。
- **列表**:差异点、应用、参数选项用 **bullet / numbered**,不埋进段落。
- **段落节奏**:每段 ≤4-5 句;拒绝连续大段墙。
- **CTA**:正文结尾**必有**一个明确的下一步(询盘 / 索取 datasheet / 配置),不能只靠页头按钮。
- **图片**:产品图放**标题之后 / 相关分节旁**,不放标题前无上下文;每图**描述性 alt**(见内容质量闸)。

## 映射到 Slate 节点(AllinCMS content 是 Slate 数组)

正文是 `[{type, children:[{text}], id}]` 的 Slate 节点数组。上面的格式落到节点:

- **分节标题** → heading 节点(`type: 'h2'` / `'h3'`),不是把标题当粗体段落。
- **加粗** → text leaf 的 `bold: true` mark。
- **列表** → 列表节点(`ul`/`ol` + `li`/`list-item`),不是「- 」开头的纯文本段落。
- **段落** → `type: 'p'`。
- 上传后前台 DOM 应能看到 `h2/h3`、`strong`、`ul/ol` 节点(见 `batch-verification.md`:凡源里的 `**bold**`/列表要转成真 Slate mark/block,别当纯文本)。

## 反例 vs 正例

**反例(段落墙,这就是那种不及格页)**:
```
[p] "High-performance phase-stable 26.5 GHz test cable holding phase within 250 PPM..."
[h2] Specifications
[表格]
[p] "P-TEST is Sinopro's phase-stable test cable ... (一大段混了描述+对比+定位)"
[p] "P-TEST fits high-volume production test stations ... (应用也是一大段)"
```
→ 无加粗、无列表、无「为什么选它/信任/CTA」分节;差异点和应用埋在段落里扫不到。

**正例(可扫读)**:
```
[p] 相位稳定 26.5 GHz 测试电缆,**-40~+85℃ 相位漂移 ≤250 PPM**,给量产测试和实验室。
[h2] 关键规格
[表格] + [p] 选型看三个:**频率 DC-26.5GHz**、**阻抗 50Ω**、**VSWR ≤1.3**。
[h2] 为什么选 P-TEST
[ul]
  [li] **相位稳定**:全温区漂移 ≤250 PPM,免反复校准
  [li] **耐用**:插拔寿命 ≥5000 次,铠装抗压
  [li] **低损耗**:...dB/m @26.5GHz
[h2] 典型应用
[ul] [li] 量产端口测试 [li] 实验室 R&D [li] 现场校准
[h2] 品质与信任
[p] SPC 内导体 + FEP 绝缘;RoHS;三年质保。
[h2] 获取报价
[p] 按料号规则配置,**点「Request a Quote」或索取完整 datasheet**。
```

## SOP:先按骨架生成,再校验——不要自由写完再补

> 关键顺序:**生成侧就把结构做对**,别自由写一坨再靠 gate 事后拦。gate 是安全网,不是作者。

1. 从 source-wiki 取 what / 差异点(≥2,带数字)/ applications / trust。
2. **跑 `build_content_skeleton.py --package <pkg> --apply`**:它按上面 6 节把每个产品/文章的 body 生成成 content-format-standard 的 Slate 骨架(hook + H2 分节 + 关键 `bold` + 差异/应用 `ul` + 结尾 CTA),**结构 correct-by-construction**。你只往骨架里填**真实 leaf 文本**,不自由排版。
3. 把每个 `[fill: ...]` 标记替换成有据可查的内容;`needs != []` 代表 source 缺字段——去补,别编造填坑。
4. 过**格式 gate**(`check_content_format.py --package <pkg>`,产品/文章正文用 `--strict`):骨架天然满足 ≥2 heading / bold / 列表 / CTA,`--strict` 把「缺分节/加粗/列表/CTA/超长段落」也升为硬拦,逼补齐。
5. 过**内容质量 gate**(`check_content_quality.py`):骨架只保证结构,填进去的文案对不对、有没有占位/幻觉,由它拦。
6. 上传后前台复验 DOM 有 `h2/strong/ul` + CTA 可见(`batch-verification`)。

## 为什么「gate 过」不等于「内容合格」——逻辑闭环

三个东西各管一段,缺一段就有漏:

| 环节 | 工具 | 保证什么 | 保证不了什么 |
|---|---|---|---|
| 生成侧 | `build_content_skeleton.py` | **结构** correct-by-construction(分节/列表/加粗/CTA 一定在) | 文案质量、分节内容是否有料 |
| 校验·格式 | `check_content_format.py`(`--strict`) | 结构缺失被拦(warn→block) | 加粗加在废话上、列表内容空 |
| 校验·质量 | `check_content_quality.py`(`--strict`) | 占位/库存图/幻觉规格/薄参数被拦 | 文案有没有说服力、分节合不合理 |
| AI 自审 + 人审 | `formatChecklist` / `aiReviewChecklist` | —— | 这才是**质量**的最后一公里 |

**诚实边界**:骨架 + 双 gate + strict 能**确保结构合规、拦住明显不合格**;但「专业质量」(分节合理性、加粗是否点在关键、文案说服力、深度)本质是判断,**任何自动闸都确保不了**,只能靠 AI 自审清单 + 人审。别把「gate 全绿」当成「内容专业」。

## 和内容质量闸的分工

- `check_content_quality`:内容**对不对**(占位/库存图/缺 alt/幻觉规格/薄参数)。
- 本规范 + 格式 gate:排版**专不专业**(层次/加粗/列表/CTA/信息架构)。
- `build_content_skeleton`:让上面两条在**生成时**就成立,而不是事后补。
- 三个都过 + 人审,才是「内容对题 + 读者扫得懂、走得到下一步」的合格页。
